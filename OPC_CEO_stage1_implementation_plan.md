## Goal

Build one complete operating loop for a one-person-company CEO:

1. Export a fixed Google Sheets operating workbook through the existing Connector without exposing the raw workbook or non-Top rows to host context.
2. Normalize, validate, diff, review, seal, and apply approved source changes into a local `OPC_WORKSPACE`.
3. Compute evidence-grounded priorities and changes since the prior approved Brief.
4. Select Top items locally with one fully specified deterministic ranking key; use Codex only to present bounded evidence, collect dispositions, and draft reviewable language.
5. Require an explicit disposition for every Top item.
6. Persist immutable Brief and decision revisions only after exact approval.
7. Make every failure visible, recoverable, testable, and auditable.

The long-term target remains five domain Skills sharing one operating contract. P0 implements only `opc-ceo-office`; the four domain Skills remain P1.

Every named range includes a fixed ordered header. Columns may contain optional blank values but may not be omitted, reordered, duplicated, or extended inside the range. Extra columns outside the range are ignored. Formulas in contract columns are invalid. Normalize text to Unicode NFC, dates to `YYYY-MM-DD`, timestamps to RFC 3339 with offset, booleans to true/false, and amounts to decimal strings plus ISO 4217 currency.

Common fields are `record_id`, `title`, `status`, `updated_at`, and optional `archived_at`. IDs match:

```text
^(priority|pipeline|receivable|contract|project|risk)_[a-z0-9][a-z0-9_-]{2,63}$
```

`workbook-contract.json` is the build-time source of truth for exact per-tab columns, enums, normalization, delta semantics, and material-change rules. Generation fixes workbook properties plus ZIP entry order/timestamps so identical contracts produce byte-identical `.xlsx` files. ## Intake Contract

Before product implementation, a host-driven capability probe must prove that the installed Connector can return requested `version`/`modifiedTime`, export a native Sheet as a materialized `.xlsx` attachment, preserve all six named ranges, expose an absolute local path, allow local Python/openpyxl to hash/read it, and state host attachment cleanup semantics. The only code permitted before this gate is an isolated `spikes/connector_capability.py` probe that creates/validates a disposable six-range workbook and writes no production package files. A passing `evidence/connector-capability-v1.json` contains no cell values and proves Connector-to-probe handoff only; it does not claim CLI integration. Failure blocks P0.

After the CLI/Skill bootstrap exists, a separate host-driven live smoke proves Connector-to-CLI stage/normalization/result behavior and writes `evidence/connector-live-v1.json`. This second receipt gates P0 acceptance/canary, not permission to begin implementation. Neither phase may fall back to reading raw cells into the host.

The Skill reads Drive metadata, exports `.xlsx`, then reads metadata again. Numeric Drive `version` is the ordering authority; `modifiedTime` is display metadata and a before/after consistency check. A changed pair discards the artifact and retries once; a lower/equal-inconsistent version is fatal. The CLI opens an external Connector temp artifact without following symlinks, validates regular-file ownership/type/size through the file descriptor, hashes while copying into a newly created staging run, and never reopens the untrusted source path.

Maintain workbook-level `last_observed_drive_version/modified_time`, `last_successful_refresh_at`, and `last_fully_applied_drive_version`. A successful metadata check advances `last_successful_refresh_at` even if the Drive version is unchanged. Source age is `now_utc - last_successful_refresh_at` in elapsed seconds, never Drive `modifiedTime` or calendar dates. Default `max_stale_hours` is 168: age `<=168h` requires explicit stale approval; age `>168h` is diagnostic-only and produces no Top or dispositions. Pending rows reuse retained normalized input; failed refresh uses applied projections only and makes no `new`/`resolved` claims.

The external Connector attachment remains host-owned. The CLI guarantees deletion only for its Workspace staging copy. The capability receipt must state the host attachment ownership and cleanup contract; if the host cannot guarantee automatic cleanup or provide a safe disposable-path contract, the capability gate fails and P0 remains blocked.

`envelope.json` records contract version, spreadsheet ID hash, before/after Drive metadata, export timestamp/hash, and a per-tab row/column/hash/completeness manifest. `complete: true` requires EOF, all six exact named ranges, bounds, and normalized hashes.

Validation severity:

| Class | Examples | Result | Apply | Deletion inference |
|---|---|---|---:|---:|
| Fatal source | partial export, version drift, missing/renamed range, out-of-order version | block run | No | No |
| Identity fatal | missing/duplicate/invalid ID, formula in identity | block run | No | No |
| Contract fatal | unsupported version, header drift, path escape | block run | No | No |
| Row quarantine | valid ID with malformed date/amount/enum/relation | preserve old row; mark degraded | Valid rows | No global deletion |
| Item conflict | three-way conflict, source missing, amount anomaly | item-level resolution | Unaffected rows | Approved tombstone only |
| Warning | optional blank, stale row, absent alias, unchanged source | continue visibly | Yes | Complete input only |

Import run and row states are separate:

```text
RUN: received -> validating -> {blocked,staged_clean,staged_degraded}
                               -> awaiting_resolution -> sealed -> applying
                               -> {applied,rejected,failed,reconciled}

ROW: valid -> approved -> applied
         \-> rejected -> pending
     quarantined ------> pending
     conflict/tombstone_pending -> approved -> applied
                                \-> rejected -> pending
```

Rows with missing, duplicate, invalid, or formula-derived IDs never enter the row state machine because they block the entire run. A quarantined row cannot be approved. A newer complete snapshot may replace pending with a valid value, another quarantined/conflict value, or a tombstone; equal-to-applied input clears pending.

Compare `SourceProjection.applied.source_hash`, incoming hash, and current canonical hash. Each projection stores independent `applied` and optional `pending` structures with source/canonical hashes, snapshot ID, Drive version, status, timestamps, and reason codes. Approval promotes pending input to applied; rejected/quarantined input remains pending without changing the stable applied base. For the same `record_id`, the latest complete observed snapshot supersedes older pending input: equal-to-applied clears pending; a different valid, quarantined, conflict, or tombstone value replaces pending; and the superseded version remains only in immutable run history. A missing record becomes a tombstone only after a complete, non-degraded read and item-level archive approval.

Amount anomaly applies only to `priority.amount`, `pipeline.amount`, and `receivable.total_amount/outstanding_amount`. Currency changes are always anomalous. Every encountered currency requires a configured threshold. If either absolute value is zero, an absolute delta above the threshold is anomalous; otherwise both a delta above threshold and a ratio of at least 3.0 are required. Normal changes may be approved as a batch. Conflicts, tombstones, and amount anomalies require item-level decisions.

Delete the Workspace raw `.xlsx` copy immediately after complete normalized output and hashes are durably verified, regardless of later review, decline, or Apply. Retain only hashes, Drive metadata, and normalized snapshots. Failed Workspace copies become cleanup-eligible at 168 hours and are deleted on the first subsequent Skill/CLI entry; without a scheduler P0 does not claim wall-clock deletion while no process runs. Cleanup failure and overdue eligibility are visible in `status` and audit. Host-owned attachment cleanup is governed only by the capability receipt.

## Candidate and Delta Contract

Source diff compares incoming source with the per-record applied projection. Brief delta compares current applied evidence with the evidence snapshot sealed into the last approved Brief.

Deterministic score:

| Component | Points |
|---|---:|
| Severity critical/high/medium/low | 40/30/15/5 |
| Overdue/today/1-3 days/4-7 days | 30/25/15/5 |
| Blocked or disputed | 15 |
| Worsened/reopened/new | 15/12/5 |
| Amount at/above approval threshold | 10 |
| Strategic weight 1-5 | 4/8/12/16/20 |

Strategic weight is the maximum of the record value, matching key-account priority, and linked objective priority; default 1. Build immutable ID, alias, key-account, and objective lookup maps once per run so validation/scoring is O(n), followed by O(n log n) sorting. Evaluate all eligible records up to the 1,000-row source limit before applying any display cap. The exact ranking tuple is `mandatory desc, score desc, severity desc/null-last, due asc/null-last, strategic_weight desc, updated_at desc, record_id asc`. Mandatory means overdue, due today, or high/critical severity. If mandatory candidates exceed 50, emit `CANDIDATE_OVERFLOW`, retain counts plus the first 50 ordered candidates for local triage, and produce no actionable Top.

Per-type inputs:

| Type | Due | Severity/order | Amount | Own weight |
|---|---|---|---|---|
| `priority` | target date | paused -> active reopens | amount | weight |
| `pipeline` | next-action due | `discovery < proposal < negotiation`; backward worsens; won/lost resolve | amount | strategic weight |
| `receivable` | due date when outstanding >0 | overdue 1-7/8-30/>30 days = medium/high/critical; dispute adds 15 | outstanding only | linked only |
| `contract` | earliest active review/signature/renewal date | risk low < medium < high < critical | none | linked only |
| `project` | milestone due | health healthy < low < medium < high < critical; blocked adds 15 | none | linked only |
| `risk` | earliest due/mitigation date | severity low < medium < high < critical | none | linked only |

The amount score is awarded at most once per candidate. For a receivable it uses only `outstanding_amount >= threshold`; `total_amount` remains relevant to import anomaly detection but never adds score.

Candidate eligibility requires score >=5 and at least one exact reason code: `SEVERITY_CRITICAL`, `SEVERITY_HIGH`, `SEVERITY_MEDIUM`, `SEVERITY_LOW`, `DUE_OVERDUE`, `DUE_TODAY`, `DUE_1_3_DAYS`, `DUE_4_7_DAYS`, `BLOCKED`, `DISPUTED`, `DELTA_WORSENED`, `DELTA_REOPENED`, `DELTA_NEW`, `AMOUNT_AT_OR_ABOVE_THRESHOLD`, or `STRATEGIC_WEIGHT_1` through `STRATEGIC_WEIGHT_5`. `Top` is exactly the first `min(3, eligible_count)` deterministic candidates. Zero candidates require no disposition; one or two require dispositions for exactly those items.

`candidate_set.json` remains local and contains all eligible candidate IDs, domain, score/components, reason codes, severity, due bucket, stale age, bounded amount/currency, delta label, strategic weight, and structured evidence IDs. It excludes Sheet/range IDs, formulas, arbitrary notes, contract text, emails, and unselected free text.

The versioned host-facing `TopApprovalView` has two manifest-controlled modes. Default `opaque` exposes run-local tokens (`T1..T3`, `E1..En`), type, enums, dates/buckets, bounded amounts/currency, score, and reason codes; it excludes canonical/source IDs, paths, title, action text, and aliases. `candidate_set.json` keeps the sealed local token-to-record/evidence mapping, and dispositions return only Top tokens. Optional `bounded_text` adds title plus these type fields: pipeline next action/account alias, receivable account alias, project milestone/owner alias, and risk owner alias. Setup requires explicit consent with a warning that person/legal names cannot be reliably detected. In bounded mode titles/actions are capped at 200 characters, aliases at 64, and emails, phone numbers, URLs, HTML, and control characters are redacted; the system does not claim legal-name removal. The full workbook, non-Top rows, contract text, arbitrary notes, formulas, and raw source identifiers never enter host context; displayed strings remain quoted untrusted data, never instructions.

## Selection, Brief, and Decisions

The local engine is authoritative for candidate eligibility, order, Top cardinality, evidence, and reason codes. Codex may explain the bounded Top view, collect structured dispositions, and draft reviewable wording, but it cannot alter Top membership, evidence, scores, or reason codes. Reject unknown/duplicate disposition IDs or fields and regenerate the review instead of silently repairing sealed input.

P0b state:

```text
drafted -> selected -> awaiting_dispositions -> sealed -> applying -> applied
              |                |                  |          |
              v                v                  v          v
           overflow         rejected           failed   reconciled
```

Every Top item requires one disposition:

- `act`: intended next action; remains visible until source resolves.
- `defer`: future review date; suppressed until then unless material source change occurs.
- `delegate`: owner alias and review date; remains in delegated status until resolved.
- `dismiss`: reason; suppressed until a material source change reopens it.

The per-type material fields and `new/changed/worsened/resolved/reopened` rules are normative in the CEO plan and workbook contract tests.

For P0b, `briefing render` canonicalizes/validates dispositions, renders the Brief, writes the four artifacts, and creates `seal.json`; it returns the exact human/JSON review plus `run_id:seal_sha256`. The user approves that displayed seal before `briefing apply` may execute. The P0b seal manifest binds sorted paths and exact bytes for `candidate_set.json`, `dispositions.json`, `brief.json`, `brief.md`, plus the current source/projection precondition hashes.

All sealed JSON uses one `canonical_json_bytes()` profile: NFC strings, UTF-8, sorted keys, no insignificant whitespace, `ensure_ascii=False`, no floats/NaN, LF, and exactly one final newline. Decimal amounts and timestamps remain strings. Candidate/Top/disposition arrays use deterministic candidate order; diagnostics sort by severity desc, code, record ID null-last, path, field; evidence sorts by record ID, path, field, source hash; reason codes use the enum order above; seal entries sort by relative path; tabs use workbook-contract order; relation IDs sort ascending. Derived Markdown uses UTF-8, LF, deterministic section order, and exactly one final newline. Normalized JSONL sorts by canonical type then `record_id` and applies the same per-line JSON profile.

Approved storage remains immutable:

```text
data/briefs/brief_YYYYMMDD_daily_rNNN.json
data/briefs/brief_YYYYMMDD_daily.current.json
briefs/daily/YYYY-MM-DD-rNNN.md
briefs/daily/YYYY-MM-DD.md
data/decisions/decision_<date>_<candidate>_rNNN.json
```

Immutable revision files are never rewritten. The explicitly mutable `.current.json` and date-only Markdown pointers are committed last and are derived/repairable. Same-day re-Apply creates a new revision.

## Apply, Audit, and Recovery

P0a import and P0b Brief use separate locks, journals, seals, approvals, and audit pairs. For each Apply:

1. Recompute seal and source/projection preconditions.
2. Acquire the per-Workspace Apply lock.
3. Stage new immutable files and pointer/projection updates.
4. Append and fsync `apply_started` with every old/new hash.
5. Replace individual files only; never claim atomic non-empty-directory replacement.
6. Commit pointers/projections last.
7. Append and fsync `apply_completed`.
8. Reconcile or quarantine every interrupted stage on the next `status`, `validate`, or Apply.

For P0a, `import resolve` canonicalizes `resolution.json`, computes the exact file/projection mutations into `apply_plan.json`, and creates `seal.json`. Its seal manifest binds sorted relative paths and exact bytes for `envelope.json`, `normalized.jsonl`, `diff.json`, `resolution.json`, and `apply_plan.json`, plus all current canonical/projection precondition hashes. The command returns the exact human/JSON review plus `run_id:seal_sha256`; only a subsequent `import apply --confirm` may mutate canonical state.

P0a, P0b, and installer each define a closed `FaultPoint` registry covering every write, fsync, replace, pointer, and completion boundary. All persistent mutations must pass through typed writer primitives in `workspace.py`/`installer.py`, and every primitive requires a `FaultPoint`. A checked-in `write-boundaries.json` independently lists expected mutation purpose, call site, writer primitive, and checkpoint. An AST test rejects direct write-mode `open`, `Path.write_*`, `mkdir`, `unlink`, `rename/replace`, and `fsync` calls outside those writer modules, then compares discovered writer calls with both the inventory and registry. Parameterized process tests fail before/after every checkpoint. This detects both an untested registered checkpoint and an unregistered mutation.

Audit stores IDs, states, versions, hashes, approvals, diagnostic codes, and timestamps, not raw sensitive cell contents.

| `opc-workspace briefing render --run RUN --dispositions FILE` | Validate every Top disposition and render deterministically. |
| `opc-workspace briefing apply --run RUN --confirm RUN:SEAL` | Commit exact Brief/decision bytes. |
| `opc-workspace status [--format human|json]` | Show source age/version, import/Brief status, quarantine, pending recovery, and audit integrity. |
| `opc-workspace backup --output PATH` | Create and validate a complete backup/export. |
| `opc-workspace connector-receipt verify --receipt FILE` | Verify host-driven metadata/export/stage evidence without retaining raw cells. |
| `opc-workspace install [--check|--update]` | Install or verify CLI/Skill without silent overwrite. |

Exit codes: `0` controlled completion including decline, `1` validation/business rule, `2` missing resource, `3` runtime/dependency, `4` unsupported version, `5` conflict/lock/source drift, `6` partial recovery, `7` approval mismatch, `8` Connector artifact/metadata contract. Every command's `--format json` returns a Schema-validated envelope with `schema_version`, `command`, nullable `run_id`, `outcome`, boolean `wrote_state`, `write_scope` (`none|staging|canonical|installation`), command-specific `data`, and ordered `diagnostics`. `wrote_state=true` means that invocation durably changed any Workspace or installation file, not merely canonical business state.

Allowed outcomes are closed per command: init `initialized|already_initialized|declined`; validate `valid|invalid|recovery_required`; import stage `staged_clean|staged_degraded|unchanged|blocked`; import review `review_ready|blocked`; import resolve `sealed|rejected|blocked`; import apply `applied|already_applied|precondition_failed|recovery_required`; briefing draft `drafted|no_candidates|stale_drafted|stale_too_old|triage_only`; briefing render `sealed|rejected|blocked`; briefing apply `applied|already_applied|precondition_failed|recovery_required`; status `healthy|degraded|recovery_required`; backup `created|failed`; connector receipt `verified|invalid`; install `installed|up_to_date|updated|drift_blocked|rolled_back|failed`. The packaged Schema defines a strict `$defs/result_<command>` data shape for each outcome with `additionalProperties:false`. The Skill must branch on outcome and Schema version, not exit code or prose; rejected P0a never enters P0b. Every nonzero result includes recovery guidance.

`SKILL.md` must locate/offer the fixed workbook template, run/check the Connector capability contract, bracket raw export with metadata, pass only artifact path and non-sensitive metadata into the CLI, invoke P0a review/approval before P0b, present only the deterministic bounded Top view, require all dispositions, and never claim legal/tax/accounting finality.

Project source is authoritative. The fresh-source bootstrap is `uv run --locked opc-workspace install`; the same installer is used after installation. ## Security and Privacy

- Connector owns credentials; no OAuth token enters Workspace.
- Connector artifacts may originate in an app temp directory but are safely opened once and copied into a fresh staging run through the validated file descriptor; all later access is inside the Workspace. Host attachment cleanup must be proven by the capability receipt.
- The host sees no raw workbook or non-Top rows. `TopApprovalView` defaults to testable opaque mode; bounded text requires explicit consent and makes no legal-name-removal claim. Every displayed string is quoted untrusted data.
- Successful Workspace raw copies are deleted after normalization; failed copies are deleted on first invocation after 168 hours. P0 makes no wall-clock cleanup claim while no process runs.
- Canonical/source/evidence IDs, paths, spreadsheet/range IDs, emails, phone numbers, URLs, arbitrary notes, contract text, formulas, and non-Top free text are excluded from host views. Opaque mode uses run-local tokens and excludes names by construction; consented bounded-text fields make no name-detection guarantee. Audit keeps hashes and local canonical IDs but no host-visible sensitive text.
- P0 is single-user/local; file permissions are the authorization boundary.

## Performance

- Maximum 1,000 nonblank source rows. Evaluate every eligible row; retain only the first 50 for overflow triage display.
- Parse `.xlsx` in read-only mode; release raw cells after normalized projection.
- Build one immutable lookup index per run; relation and strategic-weight validation is O(n), followed by O(n log n) sorting. Do not scan the full record set inside per-record loops.
- Record OS, architecture, CPU, RAM, Python version, lock hash, and power mode. Run three warmups then 20 isolated subprocess samples; compute nearest-rank p95 and peak RSS with macOS `resource.getrusage(RUSAGE_CHILDREN).ru_maxrss`, normalized to MiB. On the approved baseline machine enforce: 1,000-row normalize/validate/diff under 3 seconds and 150 MiB; 1,000-change Apply under 15 seconds and 200 MiB; Brief/status each under 2 seconds and 100 MiB.
- Measure Connector export separately; never hide network latency inside the local benchmark.
- If Drive version is unchanged, skip export/import but recompute time-sensitive candidates.
- No database or incremental index in P0. Add one only after a reproducible benchmark failure.
